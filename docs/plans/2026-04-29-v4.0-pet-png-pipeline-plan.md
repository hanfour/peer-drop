# v4.0 Pet PNG Pipeline Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use `superpowers:executing-plans` (or `superpowers:subagent-driven-development`) to implement this plan task-by-task.

**Goal:** Replace the 20K-LoC hand-coded `[[UInt8]]` sprite pipeline with a PNG-based asset system shipping all 33 Batch-2 species × 3 stages × 6 moods, while preserving existing user pets, peer-network compatibility, and iCloud sync.

**Architecture:** Bundle PixelLab-generated PNG zips inside the app, decompress + cache lazily at render time. `BodyGene` enum stays for persistence/network compat but gates render through a flat-string `SpeciesID` mapping. Mood becomes a sprite axis (6 PNG variants per species×stage) instead of an eye-overlay. Migration on first v4.0 launch is one-shot, idempotent, and preserves every existing pet.

**Tech Stack:** Swift 5.9, SwiftUI, XcodeGen, [ZIPFoundation](https://github.com/weichsel/ZIPFoundation) (new SPM dep), `CGImage` + Core Graphics for render path, NSCache for LRU.

## Source & Today's Decisions

**Source design:** `docs/plans/2026-04-27-v4.0-pet-redesign-design.md` (301 lines). **Partially superseded by today's smoke-test findings** — sections §3.2 (`SpriteDataRegistry` return type), §3.5 (pattern overlay), §4 (action animation strategy), and §2 scope (11 → 33 species) are obsolete; everything else (PetLevel migration, BodyMeta `groundOffsetY`, Classic Mode toggle, age-based promotion) carries forward.

**Decisions made 2026-04-29 (locked, do not re-debate):**
- **Q1 (c) — Backward compat:** Keep `BodyGene` enum on disk/network/cloud. At render time map old enum values (`.cat`) to flat-string `SpeciesID` (`"cat-tabby"`). Delete all 20,369 LoC in `PeerDrop/Pet/Sprites/*SpriteData.swift`.
- **Q2 (a) — Species ID shape:** Flat string namespace, e.g. `"cat-tabby"`, `"slime-fire"`, `"phoenix-light"`. Codable as plain `String`. Unknown variant → fallback to family default.
- **Q3 (b) — Mood expression: OVERLAY ICON** — *(revised from initial Q3 (c); see validation finding below).* Mood is rendered as an emoji-style icon overlay (💤 happy/sleepy/etc.) composited on top of the neutral PNG sprite. No per-mood PNG generation needed. The 321 existing zips ARE the production assets.
  > **Validation finding 2026-04-29:** Initial choice was Q3 (c) per-mood PNG variants (~1,800 new gens). 6 validation gens for `cat-tabby` happy showed PixelLab cannot preserve character identity across mood prompt variations even with mirrored color/pattern tokens — bottom row generated cats had different stripe density, different ear shapes, and even an unprompted red bow. Identity drift is fundamental to text-to-image regen. Pivoted to overlay icon approach which sidesteps the issue entirely.
- **Q4 (b) — Scope:** v4.0 ships all 33 Batch-2 species at once (no v4.1 staging).

```swift
import Foundation
import ImageIO
import ZIPFoundation

enum SpikeLoader {
    static func loadEast(zipURL: URL) throws -> CGImage {
        let archive = try Archive(url: zipURL, accessMode: .read)
        guard let entry = archive["rotations/east.png"] else {
            throw NSError(domain: "SpikeLoader", code: 1)
        }
        var data = Data()
        _ = try archive.extract(entry) { chunk in data.append(chunk) }
        guard let src = CGImageSourceCreateWithData(data as CFData, nil),
              let cg = CGImageSourceCreateImageAtIndex(src, 0, nil) else {
            throw NSError(domain: "SpikeLoader", code: 2)
        }
        return cg
    }
}
```

```swift
enum PetLevel: String, Codable, CaseIterable {
    case egg
    case baby
    case adult
    case elder

    init(from decoder: Decoder) throws {
        let raw = try decoder.singleValueContainer().decode(String.self)
        switch raw {
        case "child": self = .adult        // legacy alias
        default:
            guard let v = PetLevel(rawValue: raw) else {
                throw DecodingError.dataCorruptedError(in: try decoder.singleValueContainer(),
                                                       debugDescription: "Unknown PetLevel: \(raw)")
            }
            self = v
        }
    }
}
```

## M8: Delete Legacy Sprite Code

**Goal:** Remove the 20K LoC of `[[UInt8]]` arrays. Confirm nothing depends on them.

> **Executed as 5 phases (Plan B widget migration), 2026-05-04.** The original plan
> assumed M4.4 had migrated *all* PetSnapshotRenderer callsites — but the widget
> + Live Activity were still on the legacy stack. Phases 1-4 introduced an App

> deleted the legacy stack (~21K LoC). See `feat/v4-png-pipeline` commits
> `1e05730..f74caa2` and the M8 review-fix series for the full sequence.
>
> **Contract change introduced in phase 5:** `PetRendererV3.ultimateFallback`
> (= `cat-tabby`) now catches `SpriteServiceError.assetNotFound` and renders
> the placeholder instead of throwing. Reverses the M4-pinned ghost-throws
> contract — once the legacy widget path was gone, every consumer needed *some*
> image to avoid blank UI. SpriteService keeps the strict throw contract for
> diagnostics; renderer-layer absorbs the gap. Test renames:
> `test_render_ghost_throwsAssetNotFound` →
> `test_render_ghostBody_fallsBackToUltimatePlaceholder`.

## M9: Settings → Classic Mode Toggle — **DROPPED** 2026-05-04

**Decision:** Option (i) — drop Classic Mode entirely.

**Rationale:**
- M8 phase 5 deleted the entire legacy `[[UInt8]]` sprite stack (~21K LoC).
  Restoring a 16×16 fallback path would mean reverting that deletion or
  rewriting an equivalent — disproportionate cost for a per-user
  preference that v3.x → v4.0 user-research never validated.
- Option (ii) (per-palette PNG asset variants) would require the
  retired asset-gen sprint to run again at significant PixelLab quota
  cost — same identity-drift risk that drove the Q3 pivot from
  per-mood PNG variants to SF Symbol overlays.
- Option (iii) (runtime CIFilter contrast/saturation) is technically
  feasible but produces an "uglified" v4.0 image rather than a real
  classic look. Bad UX.

**v4.0 ships as a forced visual upgrade.** Users see new sprites on
first launch; the M10 onboarding screen frames the change positively
("your pet got a glow-up"). No Settings toggle.

**If user demand re-emerges post-ship:** treat as v4.1 work and pick
(ii) or a fresh asset variant strategy. The decision is documented
here as intentional, not deferred.

## Appendix A: ~~Asset Generation Sprint~~ — REMOVED

**Status:** Removed 2026-04-29 after validation showed PixelLab cannot preserve character identity across mood prompt variations. Replaced by **M4b mood overlay icon system** in the main milestone list.

**What was abandoned:** ~1,485 new mood-variant generations across 6 moods × 33 species × 3 stages × ~3 sub-varieties. Cost: 4–6 weeks of asset gen calendar time + 22 MB bundle bloat.

**What replaced it:** Six SF Symbol icons composited at runtime onto the existing neutral-mood PNG. Bundle stays at 3.7 MB. No new generations needed.

**Validation evidence:** 6 generations spent on `cat-tabby` happy mood (3 with wrong color due to my prompt error, 3 with corrected grey-tabby color). Even with corrected prompts mirroring the original metadata, output cats had different stripe patterns, different ear shapes, and an unprompted red bow on the adult — clearly not the "same pet expressing happiness" the user expected.

If a future v4.x wants to revisit per-mood PNG variants, alternatives to evaluate first: image-to-image edits (PixelLab if it ever supports), local pixel-art tooling for hand-drawn mood overlays, or commissioning artist-drawn mood variants.
