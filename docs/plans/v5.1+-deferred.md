# v5.1+ Deferred Architecture (post-v5.0 design)

After v5.0 ships, four architecture items from the v5 review remain. Each
is too large for a same-day session and warrants its own scoped sprint.
This doc captures the goals, complexity, and prerequisites so future-me (or
a fresh contributor) can pick any one up cold.

Items here correspond to review numbering from the post-PR-#34 deeper
review (commits 6904a35, 94567d0, c189e10, c82e173, 95b68df, a6e0b43,
bb8057d, a74cc36 covered items 1, 7, 8, 6alt, 5, 2, 3, 13).

---

## Item #4 — CADisplayLink-driven animation ✅ DONE (2026-05-11)

**Implementation:**
- `PetAnimationController` lost its `Timer`. `advance(dt:)` accumulates and
  steps frames; `paused: Bool` flag short-circuits while suspended.
  Nanosecond epsilon in the divide-floor avoids IEEE-754 drift (e.g.
  `Int(0.7 / 0.1)` returning `6` instead of `7`).
- `FloatingPetView.physicsStep(dt:)` calls
  `engine.animator.advance(dt: TimeInterval(dt))` at the top of the tick —
  before the drag/exit/enter guards, matching the Timer's pre-existing
  "fire regardless of physics state" semantics.
- `Timer === firingTimer` identity guard removed; the @MainActor reentrancy
  problem it solved no longer exists because there's no async Task
  scheduling.

**Tests:** 9 advance-driven cases in `PetAnimationControllerV2Tests`:
accumulator boundary, zero-dt no-op, paused early-out, large-dt thermal
catch-up, modulo wrap, accumulator carry across calls, setAction clearing
accumulator, pause-resume clean restart. Lifecycle tests pass unchanged
(the public `isAnimating`/`startAnimation`/`stopAnimation` surface preserved
the same semantics via the flag).

## Item #9 — Asset coverage CI badge ✅ DONE (2026-05-11)

**Implementation:**
- `Scripts/compute_v5_coverage.sh` — counts `expectedV5Coverage` entries +
  bundled `.zip` files, emits shields.io endpoint JSON. Runs on Ubuntu (no
  Xcode needed) — cheaper than the macOS-runner approach originally sketched.
- `.github/workflows/asset-coverage-badge.yml` — fires on push to `main`
  when the bundled assets, the coverage whitelist, or the compute script
  change. Commits `v5-coverage.json` to the orphan `badges` branch.
- `README.md` — shields.io endpoint badge pointing at the badges branch.

The badge auto-updates as Phase 3 mass-gen drops new v5 zips and the
operator appends them to `expectedV5Coverage`. No manual badge maintenance.

## Item #11 — Sprite atlas (replace per-frame PNGs) ✅ PoC SHIPPED (2026-05-12)

**Implementation (v5.2 PoC):**
- `Scripts/build_atlas.py` — Pillow-based generator that takes a v3.0 zip
  and writes an augmented zip containing `atlas.png` + `atlas.json`
  alongside (or instead of, with `--strip-frames`) the per-frame PNGs.
  SHA-256 dedups byte-identical frames into a single atlas slot.
  Square-ish grid layout. Stable path-sort makes the atlas binary-
  reproducible. 16 Python unit tests cover grid math, path collection,
  dedup, layout, error paths, idempotent rebuild, strip mode.
- `PeerDrop/Pet/Sprites/SpriteAtlas.swift` — pure parser for `atlas.json`.
  Mirrors `SpriteMetadata` in role (no I/O on CGImage). 7 Swift unit
  tests cover present-vs-absent atlas, rect lookup, CGRect bridging,
  schema-version rejection, malformed-JSON throws.
- `SpriteDecoder` + `SpriteService` — atlas-aware: probe for `atlas.json`,
  decode `atlas.png` once, slice frames via zero-copy
  `CGImage.cropping(to:)`. Per-frame fallback intact for legacy zips.
  Decoupling means a partially-atlased zip still works (atlas covers
  what it covers; missing paths fall back to per-frame extract).
- `SpriteAtlasIntegrationTests` — 3 end-to-end tests with synthetic
  in-Swift atlas fixtures: rotation slicing, walk animation frame order,
  metadata-missing-action rotation fallback.

**Production status:**
- Bundle asset `PeerDrop/Resources/Pets/cat-tabby-adult.zip` converted
  to atlas-only (`--strip-frames`): **43,693 → 24,569 bytes (-44%)**.
- Test fixture `PeerDropTests/Resources/Pets/cat-tabby-adult.zip` left
  per-frame (the existing `cgImagesIdentical` helper compares dataProvider
  bytes, which falsely reports cropped slices as equal because they share
  the atlas backing — fixing that helper is a separate cleanup).
- All 42 sprite + atlas tests green; full test suite verified.

**Deferred from this PoC:**
- Atlas image cache shared across `SpriteDecoder` and `SpriteService`
  (currently each decodes `atlas.png` on cache miss; ~25KB PNG → ~250KB
  RGBA per asset, fast enough to defer).
- Mass migration: only `cat-tabby-adult` is converted. Other v5 multi-
  frame assets will atlas as they ship (Phase 3 mass-gen).
- Performance bench on iPhone 8 floor — not yet run; expected fine
  since slicing is zero-copy and the master PNG is decoded once.

**Follow-ups tracked as separate work, not blocking ship.**

## Item #12 — Per-species behavior JSON

**Status:** Deferred to v6.

**Goal:** Replace per-species `*Behavior.swift` files (CatBehavior, DogBehavior, etc.)
with a JSON-driven configuration model. New species become a JSON file, not a
Swift class.

**Note (2026-05-11):** v5.0.1 retired the Ghost species, which was the canonical
example for "custom species behavior" — its `GhostBehavior.swift` is no longer
in the codebase. The motivation here is now purely additive: lowering the cost
of adding *future* species with novel physics/action patterns. If no such species
materializes, this item may be dropped entirely rather than shipped.

**Sketch:**

```json
// PeerDrop/Resources/Behaviors/cat.json
{
  "physics": {"mode": "grounded", "gravity": 600},
  "movement": "walk",
  "actions": {
    "scratch": {"cooldownSeconds": [30, 60]},
    "groom":   {"cooldownSeconds": [60, 120]}
  },
  "chat": {"position": "onTop"}
}
```

`BehaviorJSONLoader` reads + builds a generic `PetBehaviorProvider`. Swift-side
behaviors only for species needing imperative code that escapes the schema.

**Prerequisites:**
- Resolve how to express custom action sequences (exit/enter scripts, multi-step
  behaviors) — JSON rule language vs Swift fallback path.
- Round-trip migration: rewrite each `*Behavior.swift` into its JSON twin +
  ensure identical observable behavior.

**Tests:**
- Round-trip parity: each existing per-species test suite passes against the
  JSON-loaded provider.
- New species adds: just-add-a-JSON test.

**Estimated complexity:** ~3 days — schema design, loader, behavior file
migration, parity tests.

**When to ship:** v6.0 alongside atlas (#11) — both architectural. Or drop
entirely if v5.0.x cadence reveals no need for additional custom species.

**Status:** Persistent operator action, not deferred to a code commit.

`docs/release/v5.0-submission-runbook.md`'s "After successful submission"
section. Each release cycle, after `fastlane release` + before Apple
review notice, install the TestFlight build on at least one device and
manually verify:

1. V5UpgradeOnboarding appears once for v4 users (skip for fresh installs)
2. Pet walks visibly when dragged left/right
3. Idle pet shows breathing animation after 10 sec
4. (v5.0.1) If user previously had a Ghost pet, it now renders as Cat — pet identity (name, age) unchanged

actual app group entitlements, real Console output).

## Item #13 — Local TCP transport: real TLS or app-level E2E

**Decision (2026-05-12):** Path B — Layer X3DH-style handshake + Double Ratchet over local TCP. Reuses the proven remote-mailbox crypto, eliminates the half-broken `CertificateManager`, avoids certificate-in-Settings UX, gives per-message forward secrecy.

**Status:** Phase 1 (crypto core) ✅ DONE (commit pending). Phase 2 (PeerConnection wiring) pending — design below.

### Phase 1 — Crypto core ✅ DONE

`PeerDrop/Security/Protocol/LocalSecureChannel.swift`:
- Static-DH bootstrap from each peer's existing `IdentityKeyManager` Curve25519 key
- Fresh ephemeral ratchet key per session (forward secrecy)
- Initiator/responder roles decided by lex order of identity public keys
- `Frame { version, RatchetMessage }` wire format
- 11 unit tests cover handshake, round-trip both directions, interleaved, forward-secrecy (counter advance), third-party key separation, corrupted frame, unknown version

### Phase 2 — PeerConnection wiring (next session)

**Goal:** Every local-TCP `PeerMessage` flows through `LocalSecureChannel.encrypt` / `.decrypt` automatically after handshake completes.

**Sketch:**

```swift
// PeerConnection.swift gains:
var secureChannel: LocalSecureChannel?

// On TCP connect, both peers send a single plaintext bundle:
struct HandshakeFrame { let bundle: LocalSecureChannel.HandshakeBundle }

// Handshake state machine:
//   - awaitingHandshake → send my bundle, expect peer's bundle
//   - establishingChannel → run LocalSecureChannel.establish
//   - secured → all subsequent send/recv encrypted
//   - on first inbound non-handshake frame before secured: drop + log

// sendMessage(_ message: PeerMessage):
//   if secureChannel != nil:
//       let frame = try secureChannel.encrypt(JSONEncoder().encode(message))
//       try await transport.send(rawBytes: frame)
//   else: existing plaintext path

// receive loop:
//   if frame is HandshakeFrame: complete handshake
//   else if secureChannel != nil: decrypt → decode PeerMessage
//   else: existing plaintext path (v5.0.x compat)
```

**Backward compat:** A peer that doesn't send a HandshakeFrame within the first ~3 seconds falls back to plaintext mode. This lets v5.0.x peers (which know nothing about handshakes) still interop. Once all peers are upgraded, the fallback can be removed in v5.2.

**Open questions for Phase 2:**
- Where does the handshake state machine live — `PeerConnection`, `Transport`, or a new `SecureTransport` layer?
- Should fingerprint verification UI (TrustedContactStore lookup + alarm on mismatch) be wired here or in a follow-on?
- How to surface "this channel is encrypted vs plaintext" to the UI? Padlock icon next to peer name?
- Reconnect flow: re-handshake every time (current Phase 2 plan), or persist channel state to skip handshake within a session?

**Estimated complexity:** 1-2 days for Phase 2 alone.

**When to ship:** Phase 1 lands in v5.0.3 as plumbing (not yet active). Phase 2 lands in v5.1 once handshake UX + backward-compat behavior is reviewed.

**Background:** `CertificateManager` generates a P-256 keypair and immediately tries to retrieve a `SecIdentity` from Keychain, but never creates the matching `SecCertificate`. So `identity` is always nil and `ConnectionManager` falls back to the "no-TLS" code path (`ConnectionManager.swift:1575`). Local-network chat / file transfer / clipboard payloads ride a plain TCP socket carrying serialized `PeerMessage`. Surfaced by 2026-05-12 security audit.

Remote mailbox payloads are already protected by X3DH + Double Ratchet — only the LAN/loopback path is in the clear.

**Two paths forward:**

### Path A — Real self-signed TLS + pinning

Generate a self-signed X.509 cert at first launch, add it to Keychain via `SecItemAdd(kSecClassCertificate)`, then `SecIdentityCreateWithCertificate`. NWConnection uses `NWProtocolTLS.Options` with the identity. Peers pin each other's certificate fingerprint on first contact (similar to the existing remote `TrustedContactStore.fingerprint`) — same TOFU model as the SSH-style first-contact handshake the audit's claim #4 calls out as weak.

**Pros:**
- Real wire encryption — passive eavesdropping on hostile Wi-Fi can't read messages.
- Matches user expectation that "P2P file transfer" is private to peers.
- Pinning reuses the existing trust-store schema.

**Cons:**
- Self-signed cert visible in iOS Settings → Profiles & Device Management; some users find this alarming.
- Pinning lockout if user reinstalls (lost cert → new fingerprint → other peers must re-trust).
- Keychain entitlement complications in test bundles (likely root cause of audit's claim #5 about 16 failing tests in operator's environment).

### Path B — Layer X3DH/Double Ratchet over local TCP

Reuse the same crypto stack the remote mailbox already uses. Local transport becomes "remote without the relay" — same `EncryptedEnvelope` wire shape, same prekey exchange, same per-session ratchet.

**Pros:**
- Single E2E primitive across remote + local; less crypto surface area to reason about.
- Naturally forward-secret per message; better than session-keyed TLS.
- No certificate UX surface area at all.

**Cons:**
- ~Doubles per-message overhead vs raw TLS (envelope + nonce + MAC).
- Re-runs prekey + handshake on every reconnect (TLS sessions resume; ratchet does not).
- Existing remote-mailbox crypto would need to handle "no relay in front" — small refactor.

**Sketch (path B):**

```swift
// PeerConnection.swift, where currently sends `PeerMessage` directly:
let plaintext = try encoder.encode(peerMessage)
let envelope = try doubleRatchetSession.encrypt(plaintext)
let wireFrame = try encoder.encode(envelope)
try await connection.send(wireFrame)
```

Receive side decrypts envelope to plaintext, then decodes `PeerMessage`.

**Decision needed before implementation:** is the user expectation "LAN P2P is private from the LAN" (path A or B equally satisfy)? Or is it specifically "matches the same threat model as remote chat" (path B preferred)?

**Estimated complexity:**
- Path A: 1-2 days. Cert generation + Keychain dance + pinning hook + tests.
- Path B: 2-3 days. Reuse mailbox crypto, but plumb session state through the connection lifecycle.

**When to ship:** v5.2 at earliest. Not gating any v5.0.x patch.

## Item #14 — Local-peer trust model

**Status:** Deferred — partially addressed by sanitization/caps in v5.0.3, but the underlying TOFU asymmetry stands.

**Background:** Local-network first-contact promotes an unknown peer's identity key directly into `.unknown` trust level; only a *change* in identity for an existing device id triggers UI alarm (`ConnectionManager.swift:3549`). Remote-mailbox invites go through a fingerprint gate (`ConnectionManager.swift:1200`) before any keys are stored.

For a P2P workflow this is reasonable — most local peers are visually identified (same room, same Wi-Fi). But the asymmetry is real: a hostile peer on the same LAN gets TOFU-trusted on first connect, no extra step.

**Tightening options:**
- Add a "first local peer connection" pairing screen (numeric short-code matching, like AirDrop's emoji confirmation).
- Always show fingerprint hex on first connection from any new public key, regardless of mailbox/local origin.
- Make `.unknown` trust level non-persistent — re-prompt every session until user explicitly elevates to `.trusted`.

**Estimated complexity:** ~2 days of UX work, mostly view layer + a fingerprint comparator helper.

**When to ship:** v5.2, alongside (or after) the TLS decision.
