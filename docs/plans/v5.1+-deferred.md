# v5.1+ Deferred Architecture (post-v5.0 design)

After v5.0 ships, four architecture items from the v5 review remain. Each
is too large for a same-day session and warrants its own scoped sprint.
This doc captures the goals, complexity, and prerequisites so future-me (or
a fresh contributor) can pick any one up cold.

Items here correspond to review numbering from the post-PR-#34 deeper
review (commits 6904a35, 94567d0, c189e10, c82e173, 95b68df, a6e0b43,
bb8057d, a74cc36 covered items 1, 7, 8, 6alt, 5, 2, 3, 13).

## Item #4 — CADisplayLink-driven animation (replace Timer)

**Status:** Deferred to v5.1.

**Goal:** Animation frame advancement uses the same CADisplayLink that
drives physics tick, instead of an independent `PetAnimationController`
Timer. Single source of truth for clock + dt; thermal throttle scales both
together; `Timer === firingTimer` race-fix becomes unnecessary.

**Current state:** Two clocks.
- `FloatingPetView.tick(_ link: CADisplayLink)` — physics 60 Hz
- `PetAnimationController.timer` (`Timer.scheduledTimer`) — 6 Hz walk / 2 Hz idle

The Timer-callback wraps in `Task @MainActor`, which can race past
`pause()`. Today's fix: identity guard (`timer === firingTimer`) inside
`timerTick(from:)`. Works but is awkward.

**Sketch:**

```swift
final class PetAnimationController {
    private var accumulatedDt: TimeInterval = 0

    func advance(dt: TimeInterval) {
        accumulatedDt += dt
        let frameInterval = 1.0 / Double(fps)
        while accumulatedDt >= frameInterval {
            currentFrame = (currentFrame + 1) % totalFrames
            accumulatedDt -= frameInterval
        }
    }
}

// In FloatingPetView.tick(link:):
let dt = link.targetTimestamp - link.timestamp
PetPhysicsEngine.update(&engine.physicsState, dt: dt, ...)
engine.animator.advance(dt: dt)
```

**Prerequisites:**
- Resolve how `PetEngine.setupAnimationObserver`'s `$currentFrame` Combine
  pipeline reacts when frame advance is synchronous (today it's published
  on Timer fire, async observers re-render). The new path is also
  synchronous from the @MainActor pipeline so should "just work" but needs
  verification.
- Pause/resume needs reimplementation. With CADisplayLink owned by
  `FloatingPetView`, controller `pause()` becomes "stop responding to
  advance(dt:)" rather than invalidating a Timer. Probably a `paused: Bool`
  flag on the controller.

**Tests to add:**
- `test_advance_accumulates_dt_until_frameInterval_then_advances`
- `test_advance_with_zero_dt_does_not_change_frame`
- `test_advance_at_thermal_throttle_drops_frames_proportionally`

**Estimated complexity:** 2–3 hours code + thorough manual testing.

**When to ship:** v5.1, alongside any other animation-pipeline change.

## Item #9 — Asset coverage CI badge

**Status:** Deferred. Low risk, mostly CI plumbing.

**Goal:** GitHub Actions runs `xcodebuild test -only-testing:PeerDropTests/MainBundleAssetCoverageTests`, parses the test output, updates a README badge showing `v5 coverage: N/96`. Auto-updates on every PR. Visible momentum signal during the multi-month Phase 3 rollout.

**Sketch:**

```yaml
# .github/workflows/v5-coverage-badge.yml
on:
  push:
    branches: [main]
    paths: ['PeerDropTests/Pet/MainBundleAssetCoverageTests.swift']
jobs:
  badge:
    runs-on: macos-latest
    steps:
      - uses: actions/checkout@v4
      - run: xcodebuild test -scheme PeerDrop ... | tee output.txt
      - run: |
          covered=$(grep -c 'in expectedV5Coverage' output.txt)
          target=96
          curl -X POST https://shields.io/badge -d ...
```

**Prerequisites:**
- macOS runner availability (GitHub free tier provides limited macOS minutes; might need self-hosted runner).
- README.md badge URL pattern decided.

**Estimated complexity:** 1 hour CI scaffolding + 1 hour iteration on the workflow.

**When to ship:** Anytime — separate from app version cycle. No user-facing impact.

## Item #11 — Sprite atlas (replace per-frame PNGs)

**Status:** Deferred to v6.

**Goal:** Bundle each species's animations as a single PNG atlas + a JSON
UV map, instead of N separate PNGs. Reduces bundle size (PNG header dedup),
enables memory-mapped fast load, simplifies decode.

**Why deferred:** Atlas authoring is its own toolchain investment. PixelLab
exports per-frame PNGs natively; converting requires post-process script
(`Scripts/zip-to-atlas.sh`). The win materializes only after Phase 3 when
~3,000 frames exist — currently we have ~50, so the bundle-size pain
isn't acute.

**Sketch:**

```
Resources/Pets/cat-tabby-adult.atlas/
├── atlas.png    (single image with all frames laid out)
├── atlas.json   (UV map: action × direction × frame_index → rect)
```

`SpriteService` adds `loadAtlas(species:stage:) -> CGImage + AtlasMap`,
backs `frames(for:)` with atlas slicing instead of per-frame zip extract.

**Prerequisites:**
- All v5 animation frames generated (Phase 3 complete).
- `Scripts/zip-to-atlas.sh` post-process script + tests.
- Performance-test the slicing path on iPhone 8 floor.

**Estimated complexity:** ~1 week — atlas generation, slicing path,
performance validation, migration of all bundled zips.

**When to ship:** v6.0, alongside any other architectural rewrite.

## Item #12 — Per-species behavior JSON

**Status:** Deferred to v6.

**Goal:** Replace `GhostBehavior.swift` (and the implicit "default behavior"
for other species) with a JSON-driven configuration model. New species
become a JSON file, not a Swift class.

**Current state:**

```swift
final class GhostBehavior: PetBehaviorProvider {
    let profile = PetBehaviorProfile(
        physicsMode: .floating, gravity: 0,
        canPassThroughWalls: true,
        baseSpeed: 55, ...)

    func nextBehavior(...) -> PetAction { ... }
    func chatBehavior(...) -> ChatPetAction? { ... }
}
```

**Sketch:**

```json
// PeerDrop/Resources/Behaviors/ghost.json
{
  "physics": {"mode": "floating", "gravity": 0, "passThroughWalls": true},
  "actions": {
    "flicker": {"cooldownSeconds": [5, 10], "duration": 1.0, "opacity": 0.3},
    "vanish":  {"cooldownSeconds": [60, 90], "duration": 5.0}
  }
}
```

`BehaviorJSONLoader` reads + builds `PetBehaviorProvider`. Swift-side
behaviors only for species needing custom code (none yet — ghost's
behavior is the only custom one and is fully expressible in JSON).

**Prerequisites:**
- Resolve how to express custom action sequences (e.g. ghost's exit/enter
  scripts) — JSON rule language vs Swift fallback path.
- Migration: rewrite GhostBehavior contents into ghost.json + ensure
  identical observable behavior (animator timing, action dispatch).

**Tests:**
- Round-trip parity: existing `GhostBehavior` tests run against JSON-loaded
  provider should produce identical output.
- New species adds: just-add-a-JSON test.

**Estimated complexity:** ~3 days — schema design, loader, GhostBehavior
migration, parity tests.

**When to ship:** v6.0 alongside atlas (#11) — both architectural.

## Item #10 — Operator TestFlight smoke test

**Status:** Persistent operator action, not deferred to a code commit.

This is a habit, not an implementation. Documented in
`docs/release/v5.0-submission-runbook.md`'s "After successful submission"
section. Each release cycle, after `fastlane release` + before Apple
review notice, install the TestFlight build on at least one device and
manually verify:

1. V5UpgradeOnboarding appears once for v4 users (skip for fresh installs)
2. Pet walks visibly when dragged left/right
3. Idle pet shows breathing animation after 10 sec
4. Ghost flicker visible within 5–10 sec of idle
5. Aging migration silently demotes incorrectly-elder pets (check Console.app logs for category=PetEngine)

This catches device-only regressions that simulator tests miss (fonts,
actual app group entitlements, real Console output).
