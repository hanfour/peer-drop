# v5.3.6 — App Store Reviewer Notes

**TL;DR:** Critical bug fix: cross-country pairs on v5.3.5 couldn't send chat or voice messages — every send showed a red exclamation mark and never reached the peer. Root cause was an over-strict sender-side trust gate added during v5.3's audit-#14 work. v5.3.6 lowers the threshold so cross-country (relay) users can chat normally again. Receiver-side trust prompts (the actual MITM defense) are unchanged. No new features, no new entitlements, no permission changes.

## What's new in v5.3.5 → v5.3.6

### Hotfix: relay chat send was silently failing

A real user pair on v5.3.5 reported that they could read their chat history but neither side could send new messages or voice — every send showed a red exclamation mark.

Root cause: the trust gate in `sendTextMessage` / `sendMediaMessage` required the peer's contact record to be at trust level `.linked` or higher before allowing the send. Local-Wi-Fi peers can elevate to `.linked` via the SAS pairing sheet that appears on first contact. But relay (cross-country) peers had no equivalent prompt — every send hit the gate, every message was marked `.failed`, and the receiver never got the message that would have triggered their consent prompt. Both directions silently stuck.

The gate threshold is now lowered from `.linked` to `.unknown` (any non-blocked contact). The receiver-side first-contact prompt (`pendingFirstContact`) is unchanged — it still gates the first incoming envelope from an unknown peer behind explicit user approval before any X3DH session establishes, so MITM defense remains intact on the receiver side.

This restores the chat send semantics that v5.0.x → v5.0.2 had. The change touches one comparison line in `ConnectionManager.evaluateTrustGate`:

```swift
-    return contact.trustLevel >= .linked
+    return contact.trustLevel >= .unknown
```

The other audit-#14 protections (key-change alerts, blocked-contact gating, identity public key validation) are unchanged.

No new entitlements, no new resources, no new third-party SDKs.

## How to verify

1. Update to v5.3.6 on TestFlight on two devices (e.g. iPhone in one location, iPad in another).
2. Connect via relay (enter the room code, no Wi-Fi proximity).
3. Type a chat message and send. Previously (v5.3.5) it would show a red exclamation mark immediately. Now it should send normally and arrive on the peer.
4. Send a voice message — same expectation.
5. Sanity: local-Wi-Fi peers should continue to surface the SAS pairing sheet on first contact (unchanged from v5.3.5).
6. Sanity: blocked contacts should still be unable to receive messages (unchanged).

## What didn't change

- `currentProtocolVersion` stays at 2 — v5.3.6 peers interop with v5.0–v5.3.x.
- Local-Wi-Fi end-to-end encryption + SAS pairing sheet — unchanged.
- File transfer wire format, voice call, screenshot mode, push notifications, App Attest — unchanged.
- Privacy Manifest, age rating (4+), supply regions (174 countries) — unchanged.
- Pet animations, IAP, rarity tiers — unchanged.
