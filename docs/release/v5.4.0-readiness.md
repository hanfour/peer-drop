**Recommendation:** Before strict policy activation (not blocking the binary ship, since

and a scheduled GitHub Actions job (`workflow_dispatch` + `schedule: cron`) that
runs fuzz at 100K iterations per target.

1. Generate/obtain an Ed25519 signing key pair (one-time; `dev-signing-key.json` is dev-only).

4. Run `swift tools/sign-crypto-policy.swift` to produce the signed blob.
5. Deploy to Cloudflare Worker via `wrangler secret put CRYPTO_POLICY_JSON`.
6. Verify: hit `https://peerdrop-signal.hanfourhuang.workers.dev/v2/config/crypto-policy`

This step must happen before the worker policy can strengthen beyond bundled defaults.
Since bundled defaults = legacy behavior (see item 8 below), this is not blocking
the binary ship — enforcement stays in warn/legacy mode until the operator explicitly
upgrades the worker blob.

### 8. Bundled policy = legacy (no immediate enforcement at first install)

✅ PASS — `SecurityPolicy.bundledDefault` (verified in `PeerDrop/Security/SecurityPolicy.swift`):

| Field | Value | Behavior |
|---|---|---|
| `spkExpirationBehavior` | `.warn` | Expired SPKs log metric but do NOT block X3DH |
| `opkExhaustionLegacy` | `.proceedWithoutDH4` | Legacy peers without OPK: proceed (legacy behavior) |
| `opkExhaustionStrict` | `.failClosed` | v5.4+ peers without OPK: fail-closed (C2 active) |
| `skippedKeyTTLDays` | `30` | Conservative expiry (no keys evicted faster than pre-v5.4) |
| `consumedOPKPruneWindowDays` | `90` | Wide prune window (conservative) |

Note: `opkExhaustionStrict = .failClosed` means C2 is immediately active for
v5.4-to-v5.4 sessions. This is intentional per spec §4.2 — the risk is fail-closed
for NEW sessions between two v5.4 devices when OPKs are exhausted, which is the
correct and desired behavior. Legacy devices (v5.3.x) are unaffected.

The "no immediate enforcement" goal refers to v5.4↔legacy interactions, which is
correctly governed by the `.proceedWithoutDH4` legacy policy.

Worker-blob upgrade will flip `spkExpirationBehavior` from `.warn` to `.reject` and
may tighten `spkMaxAgeDays` after the 7-day soak window per spec §8.6.

## Ship Decision

**GO — with acknowledged items.**

The two FAIL/PARTIAL items (fuzz CI coverage and crypto-file line coverage) are not
blocking the binary ship for the following reasons:

1. **Activation is decoupled from ship.** The bundled-default policy keeps v5.4 in
   warn/legacy mode for legacy peers. Strict C1 rejection (`.reject`) and any future
   threshold tightening require an explicit operator action (worker blob upgrade per
   spec §8.6). No code change is needed to defer strict enforcement.

2. **Coverage gap is in edge paths, not in the mitigations.** The sub-95% files
   (X3DH.swift at 92%, DoubleRatchet.swift at 77%, PreKeyStore.swift at 91%)
   have their core happy-path and fail-closed logic covered by property tests and
   vector tests. The uncovered lines are in error-recovery branches (FS failures,
   concurrent-init races) that do not affect the correctness of C1/C2/C3/C4 for
   normal users.

3. **Fuzz gaps are incremental risk.** `PolicyFuzzTests.swift` at 10K iterations
   covers the highest-risk parsing surface (signed-blob parsing). X3DH and
   DoubleRatchet have property tests covering the same correctness invariants. The
   100K/fuzz-CI gap adds a process risk, not a known functional defect.

4. **Backward-compat cell 4 passes.** All 4 cells in the §8.4 matrix pass as unit
   tests. The "PARTIAL" annotation is a labeling clarification (unit tests vs.
   UITests per the original spec wording), not a functional gap.

**BLOCK pending (operator actions before strict enforcement):**
- Item 3 (fuzz CI): add `X3DHFuzzTests`, `RatchetFuzzTests`, and a scheduled CI job
  before the worker-blob upgrade that enables `spkExpirationBehavior: reject`.
- Item 4 (coverage): raise `DoubleRatchet.swift` from 77% to ≥ 95% before the same
  worker-blob upgrade.
- Item 7 (worker staging): must be completed before any policy strengthening.

**Timeline:** Binary can ship now. Fuzz + coverage improvements can land in a v5.4.1
hardening sprint. Worker blob upgrade (strict activation) should not happen until
those items are resolved. Spec §8.6 7-day soak window provides natural scheduling.
