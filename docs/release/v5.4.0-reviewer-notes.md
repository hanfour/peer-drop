# v5.4.0 — App Store Reviewer Notes

**TL;DR:** Security hardening release for the cloud relay transport path (the optional cross-country mode introduced in v5.0). Four targeted mitigations address replay attacks, key-exhaustion attacks, and unbounded memory growth. A new signed-config mechanism allows security thresholds to be tuned post-launch without an App Store release. No new entitlements, no new permissions, no new data collection, no UI changes for normal users.

## What's new in v5.3.6 → v5.4.0

This release hardens the end-to-end encrypted relay path without changing the local Wi-Fi path or any user-visible features.

- **Replay and stale-key protection:** peer encryption key bundles now carry a timestamp signed by the peer's own device. When Alice connects to Bob via the cloud relay, her app verifies that Bob's key bundle was published recently (within a configurable window, default 21 days) before using it to establish an encrypted session. Old or replayed key bundles are flagged and refused.

- **Stronger forward secrecy on the relay path:** when a peer's one-time key supply is temporarily exhausted, v5.4 refuses to establish a session with reduced forward secrecy. Instead it queues a retry (up to 5 attempts, 60 seconds apart) and shows a brief banner if the session cannot be established. Once the peer's device comes online and replenishes its keys, the session proceeds normally. This prevents a denial-of-service technique that previously could silently degrade encryption quality.

- **Bounded out-of-order message buffer:** the buffer that allows out-of-order message delivery is now capped by both count (200 entries) and age (30 days). This prevents an attacker who has established a legitimate encrypted session from causing unbounded memory growth by sending specially-crafted messages.

- **Tunable security thresholds via signed config:** the thresholds for the above mitigations are delivered to the app as a signed JSON blob from the relay server, verified using a public key baked into the app binary. The operator can tighten thresholds post-launch without a new App Store submission. The default thresholds shipped in v5.4.0 are identical to v5.3.x behavior — no behavior change at install; the thresholds will be tightened gradually over the 7-day post-launch soak period.

- **Wire compatibility:** v5.4 talks to v5.0–v5.3.x peers unchanged. New fields in relay envelopes are optional and gracefully absent when communicating with older clients.

- **No UI changes for normal users.** One new banner appears only when a relay session cannot be established because the peer's one-time keys are temporarily exhausted — an edge case that most users will never encounter.

## What didn't change

- End-to-end encryption fundamentals (X3DH key agreement + Double Ratchet) — unchanged.
- Local Wi-Fi transport and SAS pairing sheet — unchanged. The relay envelope's new `protocolVersion: 1` field (v5.4 generation tag) is **independent** of `currentProtocolVersion` (which remains at 2 for the local Wi-Fi handshake) — two unrelated version mechanisms.
- File transfer, voice call, screenshot mode — unchanged.
- Push notifications, App Attest device-token authentication — unchanged.
- Tip Jar IAPs (tip.small / tip.medium / tip.large) — unchanged.
- Privacy Manifest (`PrivacyInfo.xcprivacy`) — unchanged, no new data types collected.
- Age rating (4+), supply regions (174 countries) — unchanged.
- Pet animations, rarity tiers, species catalog — unchanged.

## How to verify

1. Install v5.4.0 on two TestFlight devices (an iPhone and an iPad, or two iPhones on different networks).
2. On one device, open PeerDrop and tap "New Room" to get a room code.
3. On the second device, tap "Join Room" and enter the code. The devices connect via the cloud relay.
4. Send a text message from one device to the other. Verify normal delivery (no red exclamation mark, message appears on the recipient device).
5. Optionally send a file or voice message — same expectation.
6. Normal relay session: no banners, no prompts, no behavioral difference from v5.3.6.
