### What changed in v4.0.1

PeerDrop v4.0.1 is a **hotfix** on top of v4.0.0. It removes the egg lifecycle stage from the Pet companion feature so pets appear immediately on first launch and after upgrade. No new features, no new permissions, no new data flows.

1. **Egg stage removed.** Pet lifecycle is now baby → adult → elder. The previous "egg → baby" stage was dropped because v4.0.0 shipped without egg-stage sprite assets, which caused the pet image to disappear for v3.x users whose pets were still in the egg stage at upgrade time.

2. **Welcome reveal on first launch.** New users now see a one-time welcome screen with the species name and pet sprite the first time they open the Pet tab. Previously the egg stage delayed the reveal.

3. **Silent migration for v3.x egg-stage pets.** `PetLevel` decoder maps legacy `rawValue: 1` (egg) to `.baby` silently. All existing PetState fields (name, level progress, social log, stats) are preserved. The v4 upgrade onboarding screen adds one extra line for affected users explaining their egg has hatched.

4. **Cross-version peer compatibility unchanged from v4.0.** v4.0.1 receivers also decode `level: 1` from v3.x peer payloads as `.baby`. Verified by `PetLevelDecoderTests` and `PetPayloadCrossVersionTests`.

### How to test

No login or test account required.

1. Install on a test device and launch the app.
2. Complete onboarding (or skip if already done from a prior version).
3. Tap the "Pet" tab. A pet appears immediately with a welcome reveal — no egg-hatching wait.
4. To exercise peer interaction: pair two devices via the Connection tab (QR code or shared network), then bring pets together in the social view. Greetings appear on both devices.

### Privacy & data

- No new data collected. No tracking changes.
- `PrivacyInfo.xcprivacy` unchanged from v4.0.0.
- App Group entitlement (`group.com.hanfour.peerdrop`) unchanged.

### Encryption

- Standard TLS via URLSession/WebKit; standard SRTP/DTLS via WebRTC.
- Falls under Apple's Category 5 Part 2 Note 4 exemption.
- Submission information sets `export_compliance_is_exempt: true`.

## 2. Maintainer notes (not for ASC)

### Why this is a hotfix, not a v4.1

v4.0.0 build 1 went live 2026-05-07 17:32 Taipei (phased rollout Day 1). User reported pet egg disappeared after auto-update from v3.x. Root cause: v4.0 ship missed egg-stage sprite assets; `PetRendererV3` fallback chain hits a `cat-tabby` placeholder for any `.egg` stage pet, which manifests as visual disappearance / wrong-species sprite.

Two fix paths considered:
1. **Generate 33+ egg-stage sprite assets** — large asset work, days of design time, doesn't change the fact that the egg waiting period was already weak UX.
2. **Remove the egg lifecycle stage entirely** — silent decoder fallback, no asset work, better first-launch UX (immediate welcome reveal).

Path 2 chosen. Test count net-same as v4.0 (909 — 906 baseline + 3 new tests for the welcome-suppression gate added during final review), 4 new localization keys, one new screen (`PetWelcomeView`), one new flag (`PetWelcomeFlag`).

### Migration math

| User type | v3.x state | After v4.0.1 install | Visible behaviour |
|---|---|---|---|
| Brand-new install | n/a | `.baby` (default) | Welcome reveal on first PetTab open |
| v3.x past-egg pet | `level >= 2` | unchanged | Normal v4 upgrade onboarding |
| v3.x egg-stage pet | `level == 1` | decodes to `.baby` | V4 upgrade onboarding + "egg hatched" line |
| v3.x peer (cross-version) | `level == 1` in payload | decodes to `.baby` on v4.0.1 receiver | Pet greeting renders normally |

### Things deliberately NOT mentioned to the reviewer

- **The fact that v4.0.0 just shipped 5 days ago.** Reviewer can see version history themselves; volunteering "we just shipped a broken build" invites scrutiny.
- **Phased rollout pause decision.** Maintainer-side ASC operation; not relevant to review.
- **Internal phase numbering (Phase 1–8).** Implementation detail; reviewer doesn't care.

### If reviewer asks follow-up questions

| Question | Suggested response |
|---|---|
| "Why a hotfix so soon after v4.0?" | A subset of upgrading users saw the pet image fail to render. v4.0.1 fixes the rendering path and improves first-launch UX. |
| "What happens to existing v3.x pets in the egg stage?" | They migrate silently to the `baby` stage with all fields preserved (name, level, social log, stats). A one-time onboarding screen explains the change. |
| "Are there any data migration risks?" | No. The migration is a single-field decoder mapping (`level: 1 → .baby`) with no schema change. Idempotent and reversible by reinstall. |
