## What's new in v5.0.2

Pure bug-fix patch. The v5 multi-frame walking pipeline that shipped in v5.0 had three independent bugs that conspired to keep the sprite animator stuck on the default idle frame. Combined effect: pets moved across the screen kinematically but the walking sprite never cycled ŌĆö so a user who got a cat (50% probability) saw "my cat slides without animating its legs." This release fixes the animation chain plus a small set of related physics issues.

### Animation pipeline fixes

1. **Behavior provider now tracks the pet's species.** `PetEngine.init` constructs with a random fresh-egg pet, then the app assigns the saved/mock pet afterwards. The behavior provider was bound once at init time and never rebound, so a saved cat ended up driven by whichever random species the egg had ŌĆö manifested as a cat performing dog actions (e.g. `.dig`).
2. **Species-specific actions can now exit.** Cat/Dog/Bear/etc.'s `nextBehavior` only handled `.idle`/`.walking`/`.thrown` transitions, so picking a unique action (`.scratch`, `.groom`, `.nap`, etc.) locked the pet there indefinitely ŌĆö every species got the same exit clause.
3. **Animator pipeline observes the pet's action directly.** Was observing `physicsState.velocity` and treating zero velocity as `.idle`, but `applyWalk` is kinematic (mutates position, leaves velocity at zero), so the pipeline never triggered a `setAction(.walking)` call on the sprite animator. Now `engine.currentAction` drives the animator directly.

### Physics fixes (surfaced once animation started working)

4. **Pet bounces off side walls instead of pressing into them.** `applyWalk` clamped position at the wall but didn't flip `facingRight`, so the cat walked into the right wall and stayed there until the behavior loop happened to randomly transition away.
5. **Pet walks on the floor (above the bottom tab bar), not the middle of the screen.** Engine default-init position was `(60, 200)` because `PetEngine.init` has no screen reference; FloatingPetView now snaps y to `surfaces.ground` on appearance, and `applyWalk` continually snaps y to the floor so the cat traces the bottom bar correctly.
6. **Grounded pets land cleanly after a drag-release.** `resolveCollision` previously bounced ALL pets via `bounceRestitution = 0.3` on landings with `|dy| > 20` ŌĆö frog/slime want that (`physicsMode = .bouncing`), but cat/dog/bear/rabbit etc. ended up doing a sequence of small unintended hops. Now only `.bouncing` pets actually bounce.

### Misc

- Evolution progress UI now correctly shows age (the actual gate) instead of XP (which the engine never checked). `EvolutionRequirement` simplified ŌĆö `requiredExperience` and `socialBonus` removed (neither was consulted by `checkEvolution`).
- v5UpgradeOnboarding ship-date constant corrected from 2026-05-15 ŌåÆ 2026-05-09 (Apple approved v5.0.0 the same day it was submitted, faster than the conservative buffer).
- Swift 6 readiness: 7 strict-concurrency warnings cleaned up. No behavior change, just future-proofing.

## Why this is shippable

- **No new permissions, no new APIs, no entitlement changes.** Privacy Manifest unchanged from v5.0.1 baseline.
- **No data-loss risk.** All fixes are internal to the rendering + physics pipeline; PetState persistence schema unchanged.
- **No new features, just bug fixes.** All changes are in response to user feedback after v5.0.0 ship: "my cat slides without animating", "the pet just walks into the wall", "the drop after dragging feels unnatural", "evolution progress shows XP that doesn't seem to do anything."
- **All tests green.** 374 pet tests + 8 fixture tests for the asset-coverage helper + 17 unit tests for the PixelLab normalize script.

## How to verify

If the reviewer's pet is a cat (50% probability from `BodyGene.from()`), they'll see the cat actually walking with leg cycling along the bottom of the screen, bouncing off walls without pressing into them. If the reviewer happens to roll any non-cat species (dog/rabbit/bird/etc.), they'll see correct position movement but the sprite stays single-frame ŌĆö that's the expected v4-fallback behavior for species whose v5 multi-frame data hasn't been generated yet (Phase 3 mass-gen is the operator's ongoing weekly cadence per `docs/release/v5.0.x-cadence.md`).

If the reviewer drags the pet up and releases: pet falls under gravity and lands cleanly on the floor without bouncing (for non-bouncing species).
