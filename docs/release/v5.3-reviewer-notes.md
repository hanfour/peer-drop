## What's new in v5.3

### Push notifications now actually deliver

v5.0‚Äìv5.2 carried `UIBackgroundModes: remote-notification` and called `registerForRemoteNotifications()`, but the `aps-environment` entitlement was missing from the build. iOS silently rejected APNs registration and the device never received a push token. Background invites from peers were quietly lost; the in-app WebSocket inbox still worked while the app was foregrounded, which is why this hid for several releases.

Fix: the entitlement is now declared in `project.yml` and the regenerated `.entitlements` file. `PushNotificationManager` also gained an `authorizationStatus` + `registrationState` surface so a future Settings row can show users whether their device is actually receiving pushes.

### App Attest device authentication

Previously every iOS install carried a shared "API key" baked into the IPA's `Info.plist` ‚Äî anyone unzipping the IPA could extract it and spam the relay backend on any user's behalf.

v5.3 replaces that with Apple's App Attest. On first launch the app generates a Secure Enclave-bound P-256 keypair via `DCAppAttestService`, sends an attestation to the backend (which verifies the chain against Apple's App Attestation Root CA), and trades it for a 15-minute bearer token. Subsequent calls refresh via cheap assertion exchanges; the Secure Enclave key never leaves the device.

During the transition the backend accepts both new bearer tokens and the legacy key, so v5.0‚Äìv5.2 users keep working until they upgrade.

### Local-Wi-Fi pairing sheet polish

The verification sheet that appears on first contact with a new local-Wi-Fi peer (introduced in v5.2 for the audit-#14 fix) now serializes correctly when more than one trust check fires at the same time ‚Äî previously SwiftUI logged "only presenting a single sheet" and queued the second one silently. Same UX, no more misleading log.

The sort menu in the "Nearby" tab also stops painting an empty SF Symbol slot for unselected sort modes (it used `Picker` instead, which iOS renders natively with the leading checkmark).

### Internal cleanup that's user-visible only via the changelog

- `/debug/report` (the worker route the app sends error reports to) no longer accepts arbitrary JSON or stores requester IP / User-Agent. Strict schema, 8 KB cap, fields beyond the allowlist are dropped.
- The dev tooling around the worker (npm dev deps) was upgraded to clear 10 audit vulnerabilities including four high-severity (devalue, esbuild, undici); production runtime is unchanged.
- iOS unit-test suite went from 16 failing ‚Üí 0 failing after adding `keychain-access-groups` to the test target's entitlements.

## Why this is shippable

- No new entitlements beyond `aps-environment` (push) and `keychain-access-groups` (already required for shipping secure-channel features in v5.1+).
- No new data collection. App Attest keys are local-only, never sent to PeerDrop's servers.
- Backward-compatible: v5.0‚Äìv5.2 clients keep working unchanged during the transition window.
- App Attest is Apple's first-party framework with no third-party SDK required.

## How to verify

1. Background the app on a fresh install. Have a paired peer send an invite from another device. The push should arrive within a few seconds.
2. Trigger a relay-mode connection. Console logs should show `APNs token: <8 hex chars>...` followed by `Device registered with worker` instead of the previous silent failure.
3. Open Settings ‚Üí Notifications. The new "Push Notifications" row should show a green dot + token prefix once registration completes.

## What didn't change

- `currentProtocolVersion` stays at 2 ‚Äî v5.3 peers interop with v5.0.x / v4.x.
- Local-Wi-Fi end-to-end encryption + SAS pairing from v5.2 ‚Äî unchanged.
- Pet system, file transfer wire format, voice call, screenshot mode ‚Äî unchanged.
- Privacy Manifest, age rating (4+), supply regions (174 countries) ‚Äî unchanged.
