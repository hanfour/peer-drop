## What's new in v5.2

This release combines two security tracks. v5.1 (LocalSecureChannel) was code-complete but never submitted; its features ship together with the new v5.2 trust UI.

### v5.1 features — local-Wi-Fi end-to-end encryption

Previously PeerDrop authenticated peers via a Curve25519 identity exchange during hello, but chat / files / voice signaling flowed in cleartext over the TCP socket after that. v5.1 wraps every non-control PeerMessage in a Double Ratchet session (same construction as Signal: X3DH-style bootstrap, per-message forward-secret keys). The handshake is fully automatic — kicks off when both sides finish hello.

Wire additions: `PeerIdentity.supportsSecureChannel` capability flag (v5.0.x peers decode it as false via custom Codable decoder), `MessageType.secureHandshake` (plaintext key exchange), `MessageType.secureEnvelope` (ciphertext wrapper for business messages). If handshake doesn't complete within 5s, the connection falls back to plaintext rather than hanging.

### v5.2 features — explicit pairing consent (audit-#14)

Prior to v5.2, the first time two v5.1+ devices met on local Wi-Fi the unknown peer was silently auto-trusted, assuming local-network presence implied user intent. v5.2 closes that gap.

When a fresh LocalSecureChannel handshake completes against an unknown peer, the user sees a verification sheet showing a 6-digit Short Authentication String (SAS). Both devices show the same code (derived deterministically from the sorted identity key pair via SHA-256, domain-separated with kdfInfo). The user compares the digits face-to-face and taps:
- **Accept & Trust** — elevates `.unknown` → `.linked`, the lock chip turns green.
- **Reject** — marks the peer as blocked and disconnects.

Until the user resolves the sheet, the **trust gate** rejects user-initiated file/chat/media sends to that peer. Internal protocol messages still flow so connection state stays accurate. v5.0.x legacy peers bypass the gate to preserve interop.

The longer SHA-256 fingerprint is still available via a "Show fingerprint" disclosure for power users.

## Internal performance work

`SpriteService` and `SpriteDecoder` now slice pet sprite frames from a single packed PNG atlas instead of extracting per-frame PNGs. Pure perf win, no behavior change. First migrated asset (cat tabby adult) shrank 43,693 → 24,569 bytes (-44%).

## Why this is shippable

- No new permissions or entitlements. SAS reuses the existing identity keys.
- No new data collection. All trust state stays on-device.
- Backward-compatible: v5.2 talking to v5.0.x sees `supportsSecureChannel: false` and stays on the prior path.
- Pure local computation.

## How to verify

1. Connect two v5.2 devices on the same Wi-Fi. A modal sheet appears on both showing the same 6-digit code. Tap "Accept & Trust" → green lock chip; file/chat works.
2. Tap "Reject" on either device → peer is disconnected and blocked.
3. Before approving, try sending a file → fails with "This peer hasn't been verified yet."
4. Connect a v5.2 to a v5.0.x device → no sheet (legacy fallback), transfers work as before.

`currentProtocolVersion` stays at 2 — v5.2 peers interop with v5.0.x / v4.x. Privacy Manifest, age rating (4+), data-collection profile, supply regions (174 countries) all unchanged.
